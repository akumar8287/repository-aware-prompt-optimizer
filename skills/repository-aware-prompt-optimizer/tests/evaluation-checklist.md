# Evaluation Checklist

Use this checklist to evaluate any prompt generated by the repository-aware-prompt-optimizer skill before using it with Claude Code.

Score: Pass / Fail / Partial for each item.

---

## Section 1: Clarity

- [ ] **Task is one sentence.** The Task section states exactly what must be done in one clear sentence.
- [ ] **Objective is one sentence.** The Objective states what "done" looks like — testable, not abstract.
- [ ] **Task type is labeled.** The Task Type section names one of the 9 defined categories.
- [ ] **No ambiguity in scope.** It is clear what is in scope and what is not.
- [ ] **No jargon without context.** If technical terms are used, they are specific to this codebase or explained.

---

## Section 2: Preserved Intent

- [ ] **Original request is honored.** The optimized prompt accomplishes what the user asked — nothing more, nothing less.
- [ ] **No feature creep.** The prompt does not add features the user did not request.
- [ ] **No scope reduction.** The prompt does not silently drop part of the user's request.
- [ ] **Assumptions are disclosed.** If intent was inferred, the Assumptions section is present and states what was inferred.
- [ ] **Compound tasks are split.** If the original prompt contained multiple unrelated requests, they are split into separate prompts.

---

## Section 3: No Invented File Paths

- [ ] **No absolute paths invented.** The Inspect First section does not contain absolute paths like `/home/user/project/src/auth/login.ts` unless the user provided them.
- [ ] **Paths are patterns, not assertions.** File references use patterns like `auth.controller.ts`, `src/auth/`, or "the route handler for POST /login" — not invented absolute paths.
- [ ] **User-provided paths are preserved.** If the user mentioned a specific file, it appears in the Inspect First section.
- [ ] **"Likely" language is used when uncertain.** When a file is inferred rather than stated, the prompt says "e.g.", "likely", or "such as".

---

## Section 4: Useful Inspect-First Section

- [ ] **3–6 items present.** The Inspect First section has between 3 and 6 bullet points.
- [ ] **Items match the task type.** Files listed match the investigation pattern for the detected task type (see `references/investigation-patterns.md`).
- [ ] **Most likely file is listed first.** The ordering reflects confidence — most relevant file first.
- [ ] **Config/env files included when relevant.** For deployment, auth, or payment tasks, environment variable files are listed.
- [ ] **No irrelevant files listed.** Test files, documentation files, and migration history files are not listed unless directly relevant.

---

## Section 5: Token-Saving Behavior

- [ ] **Standard constraints block is present.** All 9 standard constraint lines appear verbatim.
- [ ] **"Do not scan entire repo" is first.** The scope constraint is the first bullet in Constraints.
- [ ] **Search-before-read is instructed.** Investigation Order includes a grep or glob step before reading a file.
- [ ] **Scope expansion is conditional.** The prompt instructs to expand search only if the root cause is not found in the listed files.
- [ ] **Stop condition is clear.** Claude Code knows when to stop reading files (e.g., "stop once the root cause is found").
- [ ] **No speculative file reads.** The prompt does not say "read all files in X folder" or "check all models".

---

## Section 6: Safety Constraints

- [ ] **Task-specific constraints are present.** Below the standard block, constraints specific to this task type are included.
- [ ] **Unrelated logic is protected.** The prompt states what must NOT be changed.
- [ ] **Refactor is blocked unless requested.** If the user did not ask for refactoring, the constraints block it explicitly.
- [ ] **Dependencies are blocked unless necessary.** The prompt states not to add packages unless required.
- [ ] **Critical systems are named.** For auth/payment/data tasks, the prompt names which critical system must not be touched (e.g., "do not change token signing logic").

---

## Section 7: Copy-Paste Readiness

- [ ] **No placeholders.** No `[TODO]`, `[fill this in]`, `[your value here]`, or incomplete sections.
- [ ] **Markdown renders correctly.** Headers, bullet lists, and numbered lists are valid markdown.
- [ ] **Prompt is self-contained.** A developer can copy the entire prompt and paste it into Claude Code without needing to edit it first.
- [ ] **Expected Output section has exactly 4 items.** Root cause/approach, files changed, code for changed files, testing instructions.
- [ ] **Verification section has ≥2 concrete steps.** Each step says exactly what to do and what to observe — not just "run the app".

---

## Section 8: Unclear Prompt Handling

*(Only apply this section if the original prompt was classified as Unclear.)*

- [ ] **Clarification prompt generated instead of implementation prompt.** The output is a clarification prompt, not a fake implementation.
- [ ] **Clarification prompt is domain-specific.** If the domain is known (e.g., payments), the clarification asks targeted questions, not generic ones.
- [ ] **Clarification prompt is copy-paste ready.** The developer can paste it directly into Claude Code.
- [ ] **No implementation is attempted.** The output does not guess at a fix for an unclear request.

---

## Scoring

| Score | Meaning |
|---|---|
| All pass | Prompt is ready to use |
| 1–3 fail | Review and fix failing items before use |
| 4+ fail | Regenerate the prompt with more context |
| Any "Invented file paths" fail | Regenerate — invented paths are a hard failure |
| Any "Preserved intent" fail | Regenerate — wrong task scope is a hard failure |

---

## Quick Sanity Test

Ask these three questions before using any generated prompt:

1. **Would Claude Code know where to start reading?** (If Inspect First is vague or empty → fail)
2. **Could Claude Code accidentally break something unrelated?** (If Constraints block is missing task-specific rules → fail)
3. **Would I know if the change worked?** (If Verification is abstract, like "test it" → fail)

If any answer is "no", improve those sections before using the prompt.

---

## Section 9: Approval Preview Flow

- [ ] **Preview is shown before implementation.** The skill outputs the full Approval Preview block before any implementation begins.
- [ ] **Token comparison table is present.** The preview includes a table with original prompt tokens and optimized prompt tokens, both prefixed with `~`.
- [ ] **Next Action prompt is present.** The preview ends with the `y` / `e` / `c` options clearly listed.
- [ ] **No implementation before approval.** The skill does not write code, read files, or make changes before the user replies `y` or `yes`.
- [ ] **`y` / `yes` continues correctly.** After approval, the skill proceeds with the optimized prompt and its constraints intact.
- [ ] **`e` / `edit` asks for changes.** After `e` or `edit`, the skill asks what to change — it does not guess.
- [ ] **Edited prompt is previewed again.** After the user describes the edit, the skill regenerates the full preview block (including updated token table) and asks for confirmation again.
- [ ] **`c` / `cancel` stops execution.** After `c` or `cancel`, the skill outputs "Cancelled. Nothing was executed." and stops.
- [ ] **Unclear reply asks for y/e/c.** If the user replies with anything other than `y`, `yes`, `e`, `edit`, `c`, or `cancel`, the skill asks them to choose one of those three options. It does not infer intent.
- [ ] **Edit loop can repeat.** The preview + edit + re-preview cycle can repeat multiple times until the user confirms or cancels.

---

## Section 10: Post-Task Token Optimization Report

- [ ] **Report appears after implementation.** The token optimization report is shown after the task implementation is complete.
- [ ] **Report is not shown after cancel.** If the task was cancelled, the report must not appear.
- [ ] **Prompt token comparison is present.** The report includes the same original vs optimized token table from the preview.
- [ ] **All token numbers are prefixed with `~`.** No token figure appears without the `~` prefix.
- [ ] **Search strategy is classified.** The report states Targeted, Expanded, or Broad — not left blank.
- [ ] **Files inspected count is present.** A `~N files` figure is given, even if estimated.
- [ ] **Search operations count is present.** A `~N searches` figure is given, even if estimated.
- [ ] **Confidence level is stated.** Report says High, Medium, or Low — not a range or "unknown."
- [ ] **Confidence is explained.** One sentence explains why that confidence level was assigned.
- [ ] **No exact savings claims.** The report does not say "you saved exactly X tokens" or "reduced usage by X%."
- [ ] **No billing claims.** The report does not reference Claude Code billing, API costs, or credit usage.
- [ ] **Impact explanation is present.** At least one plain-language sentence explains why the optimized approach likely reduced total tokens.

---

## Combined Scoring

| Score | Meaning |
|---|---|
| All sections pass | Prompt and flow are production-ready |
| Sections 1–8 pass, Sections 9–10 partial | Core prompt is good; review flow behavior |
| Any Section 9 hard fail (no preview, no stop before approval) | Do not use — approval flow is broken |
| Any Section 10 exact-savings claim | Fix immediately — false claims erode trust |
